**Environment:** Ubuntu 24.04, GCC 13.3.0, C++20, CMake 3.28.3 + Ninja, vcpkg (x64-linux)

---

## 1. Introduction

### 1.1 What This Phase Accomplishes

Phase 5 delivered the Extreme Points placement engine — a function that, given a container and a sequence of item types, places them one at a time using a priority-sorted list of candidate positions. Phase 6 wraps that engine inside a full multi-objective evolutionary optimizer: the Non-dominated Sorting Genetic Algorithm II (NSGA-II). The GA evolves permutations of residual item-type indices (chromosomes), evaluates each chromosome by decoding it into a `PackingSolution` via the EP engine, and applies NSGA-II selection pressure to push the population toward solutions that minimise container count, maximise volume utilization, and minimise wasted space.

By the end of Phase 6 the program can:

- **Decode a chromosome** into a `PackingSolution` — `Packer::decode` iterates the chromosome in order, calling `ExtremePointEngine::placeItem` once per individual item, advancing to the next container (or opening a new one) when the current container's EP list is exhausted.
- **Seed 10 deterministic individuals** — the five sorting criteria from Table 5 of the paper (weight, quantity, base area, volume, volume×quantity) each produce one ascending-order and one descending-order chromosome.
- **Fill the remaining 90 slots** with shuffled permutations generated by `std::mt19937`.
- **Evaluate three objectives** for each individual: f₀ = container count (minimize), f₁ = −avg volume utilization (minimize, i.e. maximize utilization), f₂ = total wasted volume in mm³ (minimize). Infeasible individuals (items that exceed a single pallet's capacity) receive a 1000× additive penalty on all three objectives.
- **Assign Pareto ranks** via fast non-dominated sorting (Deb et al. 2002): rank 0 = Pareto front, rank k = dominated by exactly k fronts.
- **Compute crowding distance** per Pareto front: boundary individuals get +∞; interior individuals get the normalized sum of neighbour gaps across all objectives.
- **Select the next generation** via Mu+Lambda: 15 parent pairs produce 30 offspring (crossover + mutation); the 130-individual combined pool is sorted by (rank asc, crowding desc) and trimmed back to 100.
- **Apply single-point crossover with repair** for permutation chromosomes: exchange suffixes between two parents, then replace each duplicate gene in the suffix with the corresponding missing gene.
- **Apply swap mutation**: select two distinct positions at random and swap them.
- **Detect stagnation**: track the best rank-0 container count across generations; stop early if it does not improve for 5 consecutive generations.
- **Run the full GA loop** for up to 30 generations, returning the final sorted population.
- **Extract the Pareto front** from the final population for downstream solution selection.

### 1.2 Design Decisions

**`Packer::decode` introduced in Phase 6 rather than Phase 7.** The roadmap lists `Packer::decode` under Phase 7, but the GA cannot evaluate fitness without it. Creating `Packer.h/.cpp` here keeps Phase 6 self-contained and directly testable; Phase 7 builds on it without any structural change.

**Three namespaces, not one.** The GA logic is split across `Packer`, `GeneticOperators`, and `NSGA2` rather than placed in a single monolithic file. Each namespace has a single well-defined responsibility: decoding, chromosome-level operators, and selection/evaluation respectively. This mirrors the idiom used throughout the codebase (`LayerGenerator`, `BlockBuilder`, `ExtremePointEngine`) and keeps each file under 200 lines.

**`std::mt19937` passed by reference.** Every stochastic function (`fillRandom`, `crossover`, `mutate`, `tournamentSelect`, `run`) accepts its RNG by reference rather than maintaining a file-level global. This makes tests deterministic via a fixed seed (42 in the test suite) and allows the caller in `main.cpp` to seed from `std::random_device` for production runs.

**Seeded individuals are built with a template lambda inside `makeSorted`.** Five sort criteria share identical logic (sort the chromosome by a key derived from `ItemType`, ascending or descending). A single `makeSorted<KeyFn>` template function accepts any callable that maps `const ItemType&` to a comparable value, avoiding five near-identical sort blocks. The volume×quantity key uses `-> long long` explicitly because `volume() * q` can reach ~1.3×10¹¹ for large items (overflowing `int`).

**Crossover repair scans the suffix left-to-right.** After concatenating parent prefixes and suffixes, genes that appear in the prefix are counted. Any gene in the suffix whose count exceeds 1 is a duplicate; it is replaced with the next missing gene from a pre-built list. Left-to-right scanning ensures the repair is deterministic and O(n) after the initial O(n) count pass.

**`just_opened` flag in `Packer::decode` prevents infinite loops.** When the active container's EP list is exhausted, the decoder either advances to the next existing container or opens a new empty one. If even a fresh empty container cannot accept the item (the item is physically larger than the pallet), the `just_opened` flag causes the while loop to break immediately rather than repeatedly opening containers.

**Penalty of 1000× on all three objectives for infeasible items.** A penalty multiplier large enough to dominate any feasible objective value ensures penalised individuals are always on a dominated front. 1000 is sufficient: the maximum container count for a real order is in the tens, not the thousands.

**`muLambdaSelect` takes both arguments by value.** The function builds the combined pool by appending offspring into the parents vector (which was already moved in). Taking by value avoids an extra copy at the call site (`pop = muLambdaSelect(std::move(pop), std::move(offspring))`) and lets the function std::move the offspring elements into the combined vector without copying chromosomes.

**`assignCrowdingToAll` resets `crowding_distance` to 0.0 before computing.** Crowding distance is additive across objectives (each objective's loop adds to the current value). Without a reset, calling the function twice on the same population would double-count. The reset happens once at the top of `assignCrowdingToAll` before any per-front processing.

**Stagnation tracks `objectives[0]` cast to `int`.** Container count is always a whole number; casting the double objective to `int` for comparison avoids floating-point equality issues. The penalty term (1000× unplaced) is also an integer multiple, so the cast is exact for any real instance.

## Step 1: Implement `Packer::decode` — Task 6.1

### Purpose

`Packer::decode` is the bridge between the GA's search space (permutations of item-type indices) and the objective space (a concrete `PackingSolution`). It is the only function that the fitness evaluator calls. Everything the GA optimises flows through this single entry point.

### Algorithm

```
sol.containers = copy of seedContainers
eps[i] = ExtremePointEngine::init for container i

active = 0

for type_idx in chromosome:
    for each of residualCounts[type_idx] items:
        while not placed:
            placed = placeItem(containers[active], type_idx, eps[active])
            if not placed:
                if next existing container available:
                    active++
                else if not just_opened:
                    open new Container{}, init its EPs
                    active++
                    just_opened = true
                else:
                    // brand-new empty container also failed → item is infeasible
                    clean up empty container
                    out_unplaced++
                    break
```

**Why copy seedContainers?** Phase 1 block items already occupy some of the pallet volume. The EP engine's `init` call reads these placed items to seed the EP list, so the containers must carry their Phase 1 content into the decode. Each call to `decode` gets a fresh copy so chromosomes are evaluated independently.

**Why not go backwards?** Once `active` advances past a container, returning to it would require re-sorting the EP list and re-projecting all EPs against the newly placed items in that container. The greedy forward-only strategy is O(1) per container advance and produces solutions good enough for the GA to optimise.

**`just_opened` flag.** Without it, a physically infeasible item (larger than one pallet) would trigger an infinite loop: `placeItem` fails, a new container is opened, `placeItem` fails again, another new container is opened, and so on. The flag ensures we try each new empty container exactly once before giving up.

### Key points

- `out_unplaced` is an `int&` out-parameter, not a return-value component. The caller (`evaluateFitness`) uses it to apply an additive penalty; keeping it separate from the `PackingSolution` avoids modifying `Types.h`.
- The function always returns a non-empty `containers` vector: the initial `if (sol.containers.empty())` block guarantees at least one container before any placement is attempted.
- `eps` is a `std::vector<std::vector<ExtremePoint>>` — one EP list per container. Only the EP list for the `active` container is ever written to; the others remain at their seeded state until `active` reaches them.

## Step 2: Implement `makeSeededIndividuals` — Task 6.2

### Purpose

Ten deterministic starting points give the GA a head start: known-good orderings (e.g. heaviest-first for stability, largest-base-first to fill the floor quickly) are immediately available in generation 0 without waiting for random search.

### Paper specification (Table 5)

| # | Criterion | Direction |
|---|-----------|-----------|
| 1 | Weight (mass m) | Ascending |
| 2 | Weight | Descending |
| 3 | Quantity (q) | Ascending |
| 4 | Quantity | Descending |
| 5 | Base area (l×w) | Ascending |
| 6 | Base area | Descending |
| 7 | Volume (l×w×h) | Ascending |
| 8 | Volume | Descending |
| 9 | Volume × quantity | Ascending |
| 10 | Volume × quantity | Descending |

### Template helper

```cpp
template<typename KeyFn>
static Individual makeSorted(
    const std::vector<int>& residualTypes,
    const std::vector<ItemType>& itemTypes,
    KeyFn keyFn, bool ascending)
{
    Individual ind;
    ind.chromosome = residualTypes;
    std::sort(ind.chromosome.begin(), ind.chromosome.end(),
        [&](int a, int b) {
            return ascending
                ? keyFn(itemTypes[a]) < keyFn(itemTypes[b])
                : keyFn(itemTypes[a]) > keyFn(itemTypes[b]);
        });
    return ind;
}
```

The template deduces `KeyFn` from the lambda passed at each call site. For criteria 1–8, `KeyFn` returns `int`; for criteria 9–10 the lambda has an explicit `-> long long` return type because `volume() * q` can reach ~1.3×10¹¹ — well beyond `INT_MAX` (2,147,483,647).

**Key points:**
- Each seeded individual is a valid permutation of `residualTypes` because `std::sort` only reorders elements — it never adds or removes them.
- `objectives`, `rank`, and `crowding_distance` are left at their default-initialized values (empty / 0 / 0.0); `evaluateFitness` fills them before the first selection step.

## Step 3: Implement `fillRandom` and `initPopulation` — Task 6.3

### Purpose

Random shuffles diversify the population beyond the 10 seeded individuals. A homogeneous population (all seeded, no random variation) would converge prematurely because all individuals start on the same local search paths.

### Implementation

```cpp
void fillRandom(std::vector<Individual>& pop,
                const std::vector<int>& residualTypes,
                int targetSize, std::mt19937& rng)
{
    while (static_cast<int>(pop.size()) < targetSize) {
        Individual ind;
        ind.chromosome = residualTypes;
        std::shuffle(ind.chromosome.begin(), ind.chromosome.end(), rng);
        pop.push_back(std::move(ind));
    }
}
```

`std::shuffle` with `std::mt19937` produces a uniformly random permutation in O(n). The RNG is the same instance used throughout the GA run, so the sequence of random individuals is fully reproducible from the seed.

`initPopulation` calls `makeSeededIndividuals` first (always 10 individuals), then `fillRandom` to pad to `Config::GA_POPULATION` (100). The seeded individuals always occupy indices 0–9 of the initial population.

## Step 4: Implement `crossover` — Task 6.4

### Purpose

Single-point crossover combines the front of one parent with the back of another to create an offspring that inherits partial orderings from each. The repair step ensures the result is always a valid permutation.

### Algorithm

```
cut = uniform_int[1, n-1]
child = p1[0..cut) + p2[cut..n)
counts[gene] = occurrences of gene in child
missing = genes with counts[gene] == 0
for i in [cut, n):
    if counts[child[i]] > 1:
        --counts[child[i]]
        child[i] = missing[miss_idx++]
        ++counts[child[i]]
```

**Why scan only the suffix for duplicates?** The prefix is taken verbatim from p1, so every gene in it appears exactly once there. Duplicates can only enter through the suffix (from p2), so the repair only needs to touch indices `[cut, n)`.

**Why left-to-right in the suffix?** Processing left-to-right is deterministic and matches the natural reading order. It also tends to preserve the relative ordering of p2's genes in the suffix — only duplicates are displaced, not shifted — which retains more of p2's information.

**Key points:**
- Cut point is sampled from `[1, n-1]` (not `[0, n-1]`), so both halves are always non-empty. A cut at 0 or n would produce a clone of one parent, wasting a crossover operation.
- The `n ≤ 1` guard returns a copy of p1 unchanged, avoiding an invalid distribution range.
- `std::unordered_map<int, int>` for counts is O(n) average for the count phase; the repair scan is also O(n), giving O(n) total.

## Step 5: Implement `mutate` — Task 6.5

### Purpose

Swap mutation introduces small local changes that can escape local optima without destroying most of the chromosome's structure.

### Implementation

```cpp
void mutate(Individual& ind, std::mt19937& rng)
{
    const int n = static_cast<int>(ind.chromosome.size());
    if (n < 2) return;
    std::uniform_int_distribution<int> dist(0, n - 1);
    int i = dist(rng);
    int j = dist(rng);
    while (j == i) j = dist(rng);
    std::swap(ind.chromosome[i], ind.chromosome[j]);
}
```

**Why resample until `j ≠ i`?** Swapping a position with itself is a no-op. It wastes the mutation probability without making any change, which is equivalent to not mutating at all. Resampling guarantees the mutation does something.

**Key point:** A single swap is the minimal edit that preserves permutation validity. For a chromosome of length n, there are n(n-1)/2 possible swaps — enough diversity for the GA to explore, without the large perturbation of a random reshuffling.

## Step 6: Implement `tournamentSelect` — Task 6.6

### Purpose

Binary tournament selection provides selection pressure proportional to Pareto rank while maintaining diversity through the crowding-distance tiebreaker.

### Implementation

```cpp
const Individual& tournamentSelect(
    const std::vector<Individual>& pop, std::mt19937& rng)
{
    std::uniform_int_distribution<int> dist(0, n - 1);
    int a = dist(rng);
    int b = dist(rng);
    while (b == a) b = dist(rng);

    if (ia.rank < ib.rank) return ia;
    if (ib.rank < ia.rank) return ib;
    return (ia.crowding_distance >= ib.crowding_distance) ? ia : ib;
}
```

**Why return a `const Individual&` rather than a copy?** The returned reference points into the population vector, which is not modified during offspring generation. Avoiding a copy saves the chromosome vector allocation for each of the 30 parent selections per generation (30 × 100-gene chromosome = 3000 int copies saved per generation, 90,000 across 30 generations).

**Key points:**
- Resampling `b` until `b ≠ a` guarantees two distinct contestants. With a population of 100, the probability of a collision is 1/100, so at most 1–2 resamples are needed in expectation.
- Lower rank always wins. The crowding-distance tiebreaker only applies when ranks are equal — it preserves the spread of the Pareto front rather than converging to a single point.

## Step 7: Implement `fastNonDominatedSort` — Task 6.7

### Purpose

Pareto ranking partitions the population into fronts: rank 0 = non-dominated individuals (the Pareto front), rank 1 = individuals dominated only by rank-0, and so on. Selection uses rank as the primary key, so the algorithm directs search toward the Pareto front.

### Deb et al. 2002 algorithm

```
for each pair (i, j):
    if i dominates j: dominated_set[i].append(j)
    if j dominates i: domination_count[i]++

front_0 = {i : domination_count[i] == 0}
assign rank 0 to front_0

while current_front not empty:
    for each i in current_front:
        for each j in dominated_set[i]:
            decrement domination_count[j]
            if domination_count[j] == 0:
                assign rank+1 to j
                add j to next_front
    rank++
    current_front = next_front
```

**Dominance definition:** A dominates B if `A[k] ≤ B[k]` for all objectives k AND `A[k] < B[k]` for at least one k. All three objectives are minimized, so smaller is always better.

**Time complexity:** O(M × N²) where M = 3 objectives and N = population size (100). For 100 individuals, the pairwise loop runs 9,900 iterations per generation — negligible compared to the O(N) fitness evaluations (each of which decodes a chromosome).

## Step 8: Implement `crowdingDistance` — Task 6.8

### Purpose

Crowding distance quantifies how isolated an individual is within its Pareto front. High crowding distance means few neighbours — these individuals are preferred in selection to maintain a spread-out Pareto front rather than converging to a single point.

### Algorithm

```
for each objective k:
    sort front by objective k
    front[0].crowding  = +∞   (boundary)
    front[n-1].crowding = +∞  (boundary)
    range = front[n-1][k] - front[0][k]
    if range == 0: continue
    for i in 1..n-2:
        front[i].crowding += (front[i+1][k] - front[i-1][k]) / range
```

**Why +∞ for boundary solutions?** Boundary individuals represent extreme trade-offs (e.g., fewest containers but worst utilization). Giving them infinite crowding ensures they are always preferred over interior solutions of the same rank — the algorithm never eliminates a solution that occupies an extremal position on any objective.

**Why normalize by range?** Objectives have different magnitudes: container count is in the tens, wasted volume is in the billions of mm³. Without normalization, the wasted-volume objective would dominate the crowding computation. Dividing by the objective's range scales each contribution to [0, 1].

**`assignCrowdingToAll` helper.** Groups population by rank, resets all `crowding_distance` fields to 0.0, then calls `crowdingDistance` per front. The reset is necessary because crowding is additive (each objective adds to the running total); without the reset, a second call on the same population would produce double-counted values.

## Step 9: Implement `muLambdaSelect` — Task 6.9

### Purpose

Mu+Lambda selection combines the current population with the offspring it produced, applies a fresh non-dominated sort, and returns the best `GA_POPULATION` individuals. This is the NSGA-II generational replacement strategy.

### Algorithm

```
combined = parents + offspring          // 100 + 30 = 130 individuals
fastNonDominatedSort(combined)
assignCrowdingToAll(combined)
stable_sort by (rank asc, crowding desc)
return combined[0..GA_POPULATION)       // best 100
```

**Why `std::stable_sort`?** Within each rank-crowding group, `stable_sort` preserves the order in which elements appeared before sorting. This makes the selection step deterministic with respect to prior ordering, avoiding spurious sensitivity to implementation-internal sort pivots.

**Why pass by value?** Both `parents` and `offspring` are taken by value (the call site uses `std::move`). This lets the function `push_back(std::move(ind))` offspring elements into the parents vector without copying chromosomes — the combined pool is built via moves, not copies.

## Step 10: Implement `evaluateFitness` — Task 6.10

### Purpose

`evaluateFitness` is the only place in the GA where the chromosome is mapped to concrete physical quantities. It calls `Packer::decode` and extracts the three objectives from the resulting `PackingSolution`.

### Objectives

| Index | Expression | Meaning |
|-------|------------|---------|
| 0 | `containers.size() + unplaced × 1000` | Container count (fewer = better) |
| 1 | `−avgUtilization() + unplaced × 1000` | Volume utilization (higher = better, negated for minimization) |
| 2 | `Σ(L×W×H − Σ item volumes) + unplaced × 1000` | Total wasted space in mm³ |

**Why 1000× penalty for unplaced items?** The maximum realistic container count for any order is in the tens. Multiplying by 1000 guarantees the penalized f₀ exceeds any feasible f₀ by at least an order of magnitude, ensuring penalized individuals are always dominated by feasible ones regardless of how many containers the feasible solution uses.

**Why `double` for wasted volume?** A single pallet holds 1200 × 800 × 1400 = 1,344,000,000 mm³. Summing across several pallets can exceed `INT_MAX` (2,147,483,647). `double` handles values up to ~1.8×10³⁰ safely.

## Step 11: Implement `extractParetoFront` — Task 6.11

### Purpose

After the GA terminates, the caller needs the set of Pareto-optimal solutions to select from. This function filters the final population for rank-0 individuals.

```cpp
std::vector<Individual> extractParetoFront(const std::vector<Individual>& pop)
{
    std::vector<Individual> front;
    for (const Individual& ind : pop)
        if (ind.rank == 0) front.push_back(ind);
    return front;
}
```

`fastNonDominatedSort` must have been called on `pop` before this function. After `muLambdaSelect`, the returned population is always sorted and ranked, so the precondition is always satisfied at the end of `run`.

## Step 12: Implement the GA main loop (`run`) — Tasks 6.12 and 6.13

### Full algorithm

```
if residualTypes.empty(): return {}

pop = initPopulation(residualTypes, itemTypes, rng)
evaluate all individuals in pop
fastNonDominatedSort(pop)
assignCrowdingToAll(pop)

best_f0 = INT_MAX
stagnation = 0

for gen in 0..GA_NGEN-1:
    offspring = []
    for pair in 0..GA_MU-1:
        p1 = tournamentSelect(pop)
        p2 = tournamentSelect(pop)
        if rand < GA_CROSSOVER_RATE:
            c1 = crossover(p1, p2)
            c2 = crossover(p2, p1)
        else:
            c1 = clone of p1
            c2 = clone of p2
        if rand < GA_MUTATION_RATE: mutate(c1)
        if rand < GA_MUTATION_RATE: mutate(c2)
        offspring.append(c1, c2)

    evaluate all offspring
    pop = muLambdaSelect(pop, offspring)

    current_best = min(objectives[0] for rank-0 individuals in pop)
    if current_best < best_f0:
        best_f0 = current_best
        stagnation = 0
    else:
        stagnation++
    if stagnation >= GA_MAX_STAGNATION: break

return pop
```

**Offspring count is always exactly `GA_LAMBDA`.** Each of the `GA_MU` (15) parent pairs produces exactly 2 offspring, giving `2 × 15 = 30 = GA_LAMBDA`. The `static_assert` in `Config.h` (`GA_LAMBDA >= GA_MU`) prevents misconfiguration at compile time.

**Stagnation compares `int`, not `double`.** Container count is always a whole number. Casting `objectives[0]` to `int` avoids floating-point equality issues. The penalty term (1000 × unplaced) is also an integer multiple, so the cast is lossless for all real instances.

**Empty residualTypes returns an empty population immediately.** If Phase 1 packed every item into layers (no residuals), the GA has nothing to do. Returning early avoids an unnecessary population initialization and fitness evaluation loop.

## What changes in Phase 7

Phase 7 wires all modules into a runnable end-to-end pipeline via `main.cpp`. `main.cpp` will:

1. Parse a CSV input file path from `argv[1]`.
2. Read item types via `CSVReader`.
3. Run Phase 1 (layer generation → filter → merge → `buildBlocks`) if any item type can form a layer; otherwise skip directly to Phase 2.
4. Call `BlockBuilder::computeResiduals` to determine which items and how many remain after Phase 1. If `spawn_new_pallet` is true, add an empty `Container{}` to the containers vector before starting the GA.
5. Seed the RNG from `std::random_device`.
6. Call `NSGA2::run` with the residual types, residual counts, item types, and containers.
7. Call `NSGA2::extractParetoFront` to get the rank-0 solutions.
8. Select the best solution from the front — the paper prioritises fewest containers, with compactness (lowest f₂) as the tiebreaker.
9. Write the selected `PackingSolution` to JSON via `JSONWriter::writeJSON`.
10. Report generation count, best objectives, and elapsed time via `spdlog`.

Phase 7 also adds `Packer::decode` to the task list as Task 7.1, but that function was implemented in Phase 6 because fitness evaluation required it. Phase 7's work is purely orchestration: no new algorithmic logic is introduced.
